比较合理的分工方式是：**不要按“一个人做前端、一个人做后端、一个人写报告”粗暴分工**，而是按课程评分重点来分：**数据结构/算法核心、系统工程/AI Agent集成、测试文档/答辩表达**三条线并行。因为这门课明确强调：核心考核点是**数据结构与算法的应用**，同时要求 GUI、文件 I/O、模块化源码、可执行程序、PPT、课程设计报告和答辩展示。

下面以推荐选题 **Navigation System / 导航系统** 分工。

------

## 一、三人总体分工建议

### 成员 A：算法与数据结构负责人，也就是“技术核心负责人”

主要负责项目里最能体现《Data Structure Course Design》价值的部分。

负责内容：

1. **图数据结构设计**
   - 地图抽象为图：顶点表示地点，边表示道路。
   - 设计 `Vertex / Edge / Graph` 类。
   - 设计邻接表存储结构。
   - 支持大规模点和边的数据管理。
2. **地图随机生成算法**
   - 随机生成二维平面点。
   - 为每个点连接附近若干点。
   - 保证图连通。
   - 尽量避免不合理道路交叉。
   - 可以用 KD-Tree、网格划分、最近邻搜索等优化。
3. **最短路径算法**
   - 普通最短路：Dijkstra / A*。
   - 交通感知最短路：边权由距离变为动态通行时间。
   - 负责复杂度分析。
4. **核心算法说明**
   - 报告中的算法流程图。
   - 时间复杂度、空间复杂度分析。
   - 数据结构选择理由。

适合使用 AI Agent 的方式：

- 让 AI 辅助生成 Dijkstra、A*、KD-Tree、图连通性检查代码。
- 让 AI 帮忙检查边界情况。
- 让 AI 生成算法伪代码、复杂度分析初稿。
- 但最终算法逻辑必须由成员 A 自己理解，否则答辩容易被问倒。

------

### 成员 B：系统实现与可视化负责人，也就是“工程集成负责人”

主要负责把算法做成一个真正可运行、可展示的软件。

负责内容：

1. **GUI 界面**
   - 地图显示。
   - 鼠标点击选择起点和终点。
   - 缩放、拖拽、刷新。
   - 路径高亮显示。
   - 交通流量用颜色区分。
2. **系统模块集成**
   - 把成员 A 的图算法接入界面。
   - 把成员 C 的测试数据和文件系统接入程序。
   - 负责主程序入口和整体架构。
3. **文件 I/O**
   - 地图数据保存到文件。
   - 从文件加载地图。
   - 保存测试结果、路径结果、交通状态。
   - 满足课程对“Physical Storage / 文件读写”的要求。
4. **可执行程序打包**
   - 保证老师电脑上能运行。
   - 准备 `README`。
   - 准备演示版本。
   - 解决依赖问题。

适合使用 AI Agent 的方式：

- 让 AI 生成 GUI 框架代码。
- 让 AI 帮忙调试界面交互。
- 让 AI 生成文件读写模块。
- 让 AI 帮忙重构代码结构。
- 让 AI 检查运行错误、异常处理、代码风格。

如果用 Python，建议技术栈：

```text
Python + PyQt5 / PySide6 + networkx辅助调试 + matplotlib/Qt绘图
```

如果想更贴近传统课程设计，也可以用：

```text
Java + Swing/JavaFX
```

或：

```text
C++ + Qt
```

三人小组如果大量使用 AI Agent，我更推荐 **Python + PyQt/PySide**，开发效率最高。

------

### 成员 C：测试、文档、答辩与项目管理负责人，也就是“质量保证负责人”

这个角色不是“只写报告”，而是负责让整个项目最终能拿分。

负责内容：

1. **需求分析**
   - 明确系统功能：
     - 地图显示；
     - 地图缩放；
     - 最短路径；
     - 交通模拟；
     - 交通感知最短路径；
     - 文件读写；
     - 异常输入处理。
2. **测试设计**
   - 小规模图测试。
   - 中等规模图测试。
   - 大规模图性能测试。
   - 起点终点非法输入测试。
   - 图不连通情况测试。
   - 文件加载失败测试。
   - 交通拥堵路径变化测试。
3. **课程设计报告**
   - 统一整体报告结构。
   - 整理每个人的个人贡献部分。
   - 写软件测试、结论、参考文献。
   - 保证格式符合学校模板。
4. **PPT 与答辩**
   - 制作答辩 PPT。
   - 准备演示流程。
   - 准备老师可能会问的问题。
   - 安排三个人分别讲什么。

适合使用 AI Agent 的方式：

- 让 AI 生成测试用例表。
- 让 AI 整理报告初稿。
- 让 AI 把代码逻辑转化为流程图文字。
- 让 AI 生成答辩 Q&A。
- 让 AI 润色英文/中文报告。
- 让 AI 检查报告是否覆盖评分点。

| ---- | -------------------- | ------------------------------------------------- | -------------------------------------- |
| A    | 算法与数据结构负责人 | 图结构、地图生成、Dijkstra/A*、交通感知路径       | 数据结构设计、算法设计、复杂度分析     |

| C    | 测试与文档负责人     | 测试用例、测试记录、PPT、报告、答辩组织           | 软件测试、需求分析、项目管理、答辩材料 |

## 三、三个人都要参与代码，避免报告贡献不均

这门课报告占 40%，而且是**个人化评估**，报告中“个人贡献”必须详细、独立。 所以不要让 C 完全不写代码，否则 C 的个人贡献会显得弱。

更合理的做法是：

- A 写核心算法代码；
- B 写 GUI 和集成代码；
- C 写测试脚本、数据生成验证脚本、异常测试代码，同时维护文档。

这样三个人都有代码贡献，也都有报告内容可写。

------

## 四、最推荐的实际执行方案

三人小组最合理的安排是：

```text
A：保证“算法深度”
B：保证“系统可运行、可展示”
C：保证“测试完整、报告规范、答辩稳”
```

如果想拿高分，重点不是功能堆得很多，而是要让老师看到：

1. 你们确实用了核心数据结构；
2. 你们确实做了算法设计和复杂度分析；
3. 程序能稳定运行；
4. GUI 能清楚演示；
5. 文件 I/O 和测试记录完整；
6. 每个人都有清晰、可证明的贡献；
7. AI Agent 是辅助开发，而不是替代理解。

最稳妥的结论是：**A 负责算法，B 负责系统，C 负责质量；三人共同理解核心算法，共同参与答辩。**
